Automatic and demand-driven — "Active" is the only lever. Calls spread across the day (better pickup, safer for the number). Dial cap is 40/number/day; scale by adding numbers, not by pushing one harder.

Reactive — no daily run, no dial cap, no spam risk (the person called in). A 2am caller never hits a dead line: the agent answers, qualifies, and books straight onto the calendar. Same booking core as outbound.

Capacity throttle: the system books only as fast as the client can absorb. Open bookings count against their limit; at the limit, outbound dialing pauses. The client marking a meeting closed frees a slot and wakes the engine. No auto-close — that's the forcing function to stay engaged. Dashed boxes are optional per account.

0 The life of one contact

stateDiagram-v2 [*] --> new: entered (upload / form / scrape / inbound) new --> enriched: researched (scraping pipe only) new --> scrubbed: clean list — skip enrichment enriched --> scrubbed: passed opt-out enriched --> disqualified: failed ICP or on DNC scrubbed --> calling: picked up by dialer calling --> booked: meeting set calling --> not_interested: said no calling --> no_answer: did not pick up no_answer --> calling: retry (under 3, +3 days) no_answer --> [*]: exhausted → follow-up list booked --> [*] not_interested --> [*] disqualified --> [*]

The state column IS the queue — a DB query in priority order drives the run. Enrichment + the ICP check only apply to the scraping pipe; a clean uploaded list skips straight to the opt-out gate. Inbound calls land directly as a booked outcome.
